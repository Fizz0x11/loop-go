这是什么

loop-go 是 Hermes Agent 的 GitHub 项目处理循环。触发一次，AI 跑完全链路，不半途而废。


适用场景：想了解一个新项目 / 想把项目介绍给同事 / 想把项目沉淀进自己的知识库。

干活 agent 和验收 agent 分离，确保测试结果是真实的，不是 AI 幻觉。

核心设计：双 agent 分离

干活 agent = 执行层（clone + 测试 + Wiki 初稿）
验收 agent = 裁判层（对照标准打分，不合格打回）
两者不是同一个 agent，不能读彼此的 connector 文件。
Agent 角色 读什么 输出什么
干活 agent 执行者（maker） clone-and-test.md 真实测试结果 + Wiki 初稿
验收 agent 裁判（checker） verification.md 打分 + 结论（通过/补项/打回）

为什么要分离？ 同一个人又干活又验收，容易自己骗自己。验收 agent 有独立判断标准，测试结果必须是真实的，不能是"看起来没问题"。

验收打分标准

分数 结论 处理
8-10 分 ✅ 验收通过 进入下一步
5-7 分 ⚠️ 需补项 列出具体要补什么，回到干活 agent 补
0-4 分 🔴 不合格 打回重做

打分维度：A 项目基本信息（clone 成功？README 完整？）+ B 真实性检查（依赖真装了？测试命令真跑了？）+ C Wiki 质量（格式规范？内容完整？）

648-1 自我复盘表

Loop 跑完之后，AI 必须自检这次 loop 自身有没有敷衍。

# 检查项 合格标准
1 clone 真跑了？ gh repo clone 命令实际执行了
2 测试命令真跑了？ 依赖安装/验证命令实际执行了
3 renwei handoff 写了？ 写了一份符合格式的 handoff 追加到 BD
4 CE Review 栏填了？ 做对了/做错了/下次改 三项都填了
5 自检有没有敷衍？ 填的都是 ✅ 还是真的有原因？
6 下次改什么？ 本次最大 1 个改进点
